## Overview

`quantlib_for_maven` packages the open-source [QuantLib](https://www.quantlib.org/) C++ library as a Java artifact published to Maven Central. It uses SWIG to generate JNI bindings, CMake to compile platform-specific native libraries, and Maven to assemble everything into a single JAR with bundled native binaries for Linux (amd64, arm64), macOS (universal binary: arm64 + amd64), and Windows (amd64).

- **Maven coordinates:** `io.github.ralfkonrad.quantlib_for_maven:quantlib`
- **Current version:** 1.42.1 (tracks upstream QuantLib releases)
- **License:** BSD 3-Clause
- **Java minimum:** JDK 17; also tested on JDK 21 and 25

## Build System

### CMake (C++ / SWIG layer)

`CMakeLists.txt` is concise at 67 lines and does exactly what it needs to:

- Requires **CMake 4.0.0**, **C++17**, **Boost 1.90.0** (headers only), **SWIG 4.4.1**
- Pins exact versions for Boost and SWIG — unusual but intentional: SWIG's Java output is sensitive to minor version differences, and Boost's CMake integration changed behaviour at 3.30
- Enables `QL_ENABLE_THREAD_SAFE_OBSERVER_PATTERN=ON` — this is load-bearing; the JVM's GC runs on separate threads and will call destructors concurrently with QuantLib's observer notifications if this is off
- Disables `QL_FASTER_LAZY_OBJECTS` — trades some runtime performance for correctness under concurrent access

`CMakePresets.json` enforces Ninja as the generator and treats compiler warnings as errors. Two presets (`debug`, `release`) write to `build/{presetName}`. This is clean.

`cmake/Boost.cmake` patches around a CMake 3.30 breaking change in Boost discovery — a good example of targeted workaround with minimal blast radius.

`cmake/CreateJavaLibraryDirectory.cmake` detects the host architecture at configure time and directs native output to the correct path under `java/target/generated-resources/swig/native/`, which Maven then picks up as resources. For Linux and Windows the path includes an `{arch}` segment (`linux/amd64`, `windows/amd64`, etc.). For macOS the path is simply `macos/` with no arch segment, because `CMAKE_OSX_ARCHITECTURES "arm64;x86_64"` produces a single universal (fat) binary containing both slices.

### SWIG (`swig/`)

`QuantLibEntrypoint.i` is the project's own interface file; it includes the upstream `quantlib.i` from the `QuantLib-SWIG` submodule. Project-level customisations:

- Makes the JNI class package-private (`jniclassclassmodifiers="class"`) — prevents accidental direct use
- Adds a static initializer that calls `QuantLibJNIHelpers.loadLibrary()` before any JNI call
- Defines `QL_JAVA_INTERFACES` to inject `AutoCloseable` into every SWIG-generated type
- Adds `java.util.RandomAccess` to `std::vector` wrappers (correct — vector is O(1) access)
- Renames `IntervalPrice::close()` to `closePrice` to avoid a name clash with `AutoCloseable.close()`

The `AutoCloseable` injection is the most important design decision in the entire SWIG layer. Without it, native objects would have no deterministic lifecycle hook visible to Java code.

`swig/CMakeLists.txt` wires SWIG → JNI compilation and links `QuantLibJNI` against `ql_library`, `Boost::headers`, `Threads::Threads`, and `JNI::JNI`. It optionally invokes the Maven wrapper after the native build completes (controlled by `QL_MVN_BUILD_JAVA`).

### Maven (`java/`)

`pom.xml` at 430 lines is on the heavier side but well-organised. Notable choices:

- **`sortpom-maven-plugin`** enforces a canonical POM element order on every build — removes a whole category of noisy diffs
- **`maven-enforcer-plugin`** gates on Maven 3.8+ and JDK 17–25 — surfaces incompatible environments early
- **`build-helper-maven-plugin`** adds `target/generated-sources/swig` and `target/generated-resources/swig` to the build, connecting the CMake output to Maven without manual copying
- The `clean` lifecycle is configured to **preserve** the `generated-sources/swig` and `generated-resources/swig` trees — if you ran CMake separately, `mvn clean` will not destroy the native output
- **`central-publishing-maven-plugin`** handles Maven Central publishing
- **`maven-gpg-plugin`** is gated behind an `enable-gpg-plugin` profile — local builds don't need a GPG key
- Profiles for JDK 21 and JDK 25 adjust `maven-compiler-plugin`'s `release` setting

The Maven wrapper (`mvnw`) is committed, which is good practice for reproducible CI. The wrapper version is managed through `.mvn/wrapper/maven-wrapper.properties`.

## Java Source Code

### Native Loading (`org.quantlib.helpers`)

**`QuantLibJNIHelpers.java`** is the bridge between the SWIG static initializer and the bundled native library:

- Detects OS via `System.getProperty("os.name")` (case-insensitive prefix match for `linux`, `mac`, `win`)
- Detects architecture via `System.getProperty("os.arch")` (`aarch64`→`arm64`, `amd64`/`x86_64`→`amd64`)
- For Linux and Windows, constructs an architecture-qualified path (`/native/{os}/{arch}/lib*.so` etc.); for macOS, uses `/native/macos/libQuantLibJNI.jnilib` with no arch segment — correct because the macOS build is a universal binary
- Throws typed exceptions (`UnsupportedOperatingSystemException`, `NativeLibraryLoaderException`) rather than raw `RuntimeException` — good for callers that may want to handle these

macOS uses a single path without architecture differentiation (`/native/macos/libQuantLibJNI.jnilib`). This is correct by design: `CMakeLists.txt` sets `CMAKE_OSX_ARCHITECTURES "arm64;x86_64"`, which produces a universal (fat) binary containing both arm64 and x86_64 slices. A single file covers all macOS architectures, so no architecture subdirectory is needed and no code change is required to support Intel Macs — they are already handled.

**`NativeUtils.java`** extracts the resource from the JAR to a temp directory and calls `System.load()`. It correctly:

- Uses POSIX file permissions (owner rwx) on the temp file before loading
- On POSIX systems, deletes the file immediately after `System.load()` — valid because the kernel keeps the file descriptor open; this prevents leftover temp files on crash
- On non-POSIX systems, registers a JVM shutdown hook for cleanup
- Throws `FileNotFoundException` (not `NullPointerException`) when the resource is missing

This is a solid, well-understood pattern for bundled native libs. The code is credited to Adam Heinrich's native-utils (MIT) with modernisation.

### Module System

`module-info.java` declares a proper named module:

```java
module io.github.ralfkonrad.quantlib {
    requires org.slf4j;
```

The `org.quantlib` package is SWIG-generated and contains hundreds of classes; exporting it is required. `org.quantlib.helpers` is the project's own hand-written layer. This is a correct and minimal module descriptor.

**Strengths:**

- `EuropeanOptionTest` is the most thorough test in the suite by a wide margin: 42 reference datasets, Greek calculations (delta, gamma, theta, rho, dividendRho, vega), implied volatility inversion, and consistency checks across ten pricing engines (Binomial JR/CRR/EQP/TGEO/TIAN/LR/JOSHI, Finite Difference, Integral, Monte Carlo, QMC). This class alone provides strong confidence in the JNI binding's numeric correctness.
- All tests correctly use try-with-resources for every SWIG object. This is non-trivial discipline — any native object leaked in a test would silently accumulate until the JVM is killed.
- Modern Java idioms throughout: `var`, `java.time.LocalDate`, `@Test` (no JUnit 4 leftovers).

**Gaps:**

- No test for the native library loading path itself (e.g., mocking `NativeUtils` to verify correct path construction per OS/arch combination)
- No negative tests for the `UnsupportedOperatingSystemException` / `NativeLibraryLoaderException` paths
- `InstrumentTest` and `ZigguratXoshiro256StarStarGaussianTest` are thin — likely placeholder-level coverage
- No integration test that exercises memory under pressure (e.g., creating and closing thousands of objects in a loop to check for native leaks)

## CI/CD

### `build_maven_artefact.yml` — Main Pipeline

Runs on push, PR, and manual dispatch. Three parallel native build jobs:

| Runner | JDK | Native target |
|--------|-----|---------------|
| macOS 14 (arm64) | 17 | macOS universal (arm64 + x86_64) |
| Ubuntu 22.04 ARM | 17 | Linux arm64 |
| Windows 2022 | 17 | Windows amd64 |

The single macOS runner produces a universal (fat) binary via `CMAKE_OSX_ARCHITECTURES "arm64;x86_64"`, so one build covers both Apple Silicon and Intel Macs without requiring a separate Intel runner in this pipeline. Notably, **there is no Linux amd64 matrix job** — that binary is produced by the deploy job itself (see below).

Each matrix job uses the composite action `.github/actions/build_native_library`, uploads the `.so`/`.jnilib`/`.dll` as a 1-day-retention artifact, then a single deploy job on `ubuntu-22.04` (x86_64):

1. Downloads the macOS, Linux ARM64, and Windows native artifacts produced by the matrix
2. Sets version via `mvn versions:set`
3. Runs the same `build_native_library` composite action locally — this is what produces the Linux amd64 `.so` for the published JAR
4. Runs `mvn clean deploy -P enable-gpg-plugin` with GPG signing
5. On `master` or manual dispatch: publishes to Maven Central

This is a correct fan-out/fan-in pattern with the deploy runner pulling double duty as the Linux amd64 builder. The 1-day artifact retention is intentional — native libs are only needed within a single pipeline run.

**Observation:** Because the Linux amd64 binary is produced inside the deploy job rather than as a separately uploaded artifact, it is the only native target without an independent CI signal — a regression that only affects Linux amd64 will surface as a deploy-job failure rather than a clearly labelled matrix failure. The broader matrix in `build_native_libraries.yml` does cover `ubuntu-24.04` (amd64), so this is mostly a labelling concern, not a coverage gap.

### `build_native_libraries.yml` — Broader Matrix Testing

Tests on six runners — Ubuntu 24.04, Ubuntu 24.04 ARM, macOS 15, macOS 15 Intel, macOS 26, Windows 2025 — all on JDK 17. A separate `include:` block adds two more jobs on `ubuntu-24.04` only, using JDK 21 and JDK 25. So the JDK cross-version coverage is Linux-only; the macOS and Windows runners are not tested against JDK 21 or 25 here. Including `macos-15-intel` is still valuable: it validates that the universal binary actually runs correctly on x86_64 hardware, complementing the arm64-only production build runner.

### `build_with_quantlib_latest.yml` — Upstream Sync

Runs daily at 02:23 UTC. Updates the `QuantLib` and `QuantLib-SWIG` submodules to their latest `master` commits and creates a PR via `peter-evans/create-pull-request`. This keeps the project tracking upstream continuously rather than only at release time — a sound strategy for catching breaking changes early.

### Composite Action (`build_native_library`)

The composite action encapsulates all platform complexity:

- **Linux:** Removes system CMake/SWIG (versions too old), installs Boost/CMake/SWIG via Homebrew on Linux
- **macOS:** `brew update && brew install boost cmake ninja swig`
- **Windows:** Downloads Boost via `Start-BitsTransfer`, installs SWIG from zip, CMake/Ninja via Chocolatey, activates MSVC dev shell

The Windows setup is the most involved and most likely to break on runner image updates. The `Start-BitsTransfer` for Boost downloads a specific URL that will need updating when the pinned Boost version changes.

### Dependabot

Configured for both Maven and GitHub Actions, all on daily schedules at staggered times (01:00–03:00 Europe/Berlin). Five separate `github-actions` entries cover `/`, `/.github/actions/build_native_library`, and its three subdirectories (`env_variables`, `setup`, `setup/jdk`). One `maven` entry covers `/java`. This is thorough — composite action steps nested in subdirectories are covered explicitly rather than relying on root-level discovery.

## Design Decisions

### AutoCloseable as a First-Class Contract

The most important architectural decision in the project is injecting `AutoCloseable` into every SWIG-generated type. SWIG objects hold a C++ pointer (`swigCPtr`) that the JVM GC cannot see or follow. The JVM heap sees only a few bytes per object, so GC pressure never triggers collection of objects that may hold megabytes of C++ memory.

The solution — making every generated type `AutoCloseable` and requiring try-with-resources — is the correct approach. Alternatives (finalizers, `PhantomReference` cleaners) are non-deterministic and have been deprecated or discouraged since Java 9. The project's documentation makes this constraint explicit and the test suite models the correct pattern uniformly.

### Exact Dependency Pinning

Boost 1.90.0 and SWIG 4.4.1 are pinned exactly, not to version ranges. For a project producing binary artifacts that must be bit-for-bit consistent across runners and releases, this is correct. The cost is periodic manual bumps; the benefit is reproducible builds.

### Bundled Multi-Platform Natives

The JAR contains native libraries for all supported platforms. This is the simplest consumption model for end users — one dependency, no native installation required. The tradeoff is a larger JAR (~30–60 MB depending on compression) and the need to rebuild and republish for every new platform. The macOS binary is a universal (fat) binary built with `CMAKE_OSX_ARCHITECTURES "arm64;x86_64"`, so a single build from an arm64 runner covers both Apple Silicon and Intel Macs — no separate Intel build job or loading-path change is needed.

### Submodule Strategy

`external/QuantLib` and `external/QuantLib-SWIG` are pinned git submodules at exact release tags for production builds, but the daily sync workflow tracks `master`. This gives the best of both: stable releases from known good commits, early warning of upstream breakage.

## Issues and Observations

**Minor:**

1. **Linux amd64 has no labelled CI signal:** The Linux amd64 binary is built inside the deploy job in `build_maven_artefact.yml` rather than as its own matrix entry. A pure Linux-amd64 regression surfaces as a deploy-job failure rather than a clearly labelled matrix failure. Promoting it to a fourth matrix entry would make CI output more diagnosable at the cost of one extra build.

2. **Windows Boost download URL** in the composite action setup: It references a specific Boost version URL. When the pinned Boost version is bumped, this URL must be updated manually in two places (`CMakeLists.txt` and the action setup script). There is no automated check to keep them in sync.

3. **`code-smells.md` is unlinked from CI / tests:** The audit is maintained and useful (covers Security/CI, SWIG/JNI, CMake) and is referenced from `AGENTS.md`, but it lives entirely outside the build. None of its open items (e.g., S9 workflow-level least-privilege permissions, S10 token scope documentation, S11 AutoCloseable regression guard, P2 `QL_JAVA_INTERFACES` macro shape) currently has an associated lint rule, CI check, or tracked test. Without that, items can quietly age out of accuracy — the doc itself already records one such cleanup pass in its "Removed After Re-evaluation" section.

4. **`CONTRIBUTING.md` is a one-line stub:** The file contains a single line ("# Contribution is welcome"). For a project of this maturity, a short pointer to `AGENTS.md` / `HOWTO-BUILD-FROM-SOURCE.md` / `code-smells.md` and a sentence on the PR workflow would set clearer expectations for new contributors.

**Structural:**

5. **Test coverage asymmetry:** `EuropeanOptionTest` is comprehensive; other test classes are thin. The date, index, and RNG tests are functional sanity checks rather than systematic coverage. Adding tests for unsupported-platform error paths and native memory lifecycle (allocate → close → allocate cycle) would improve confidence.

6. **JDK cross-version coverage is Linux-only:** `build_native_libraries.yml` only exercises JDK 21 and 25 on `ubuntu-24.04`. macOS and Windows are tested on JDK 17 only. Any JDK-version-specific behaviour in the JNI loading or in `java.time` bridging would not be caught on those platforms until a downstream consumer reports it.

## Summary

`quantlib_for_maven` is a well-engineered, production-quality open-source project. The core design — SWIG-generated JNI bindings with injected `AutoCloseable`, bundled multi-platform natives, and Maven Central publishing — is sound and correctly implemented. The CI/CD is comprehensive, the documentation is thorough, and the test suite demonstrates correct usage patterns.

The most important correctness constraint (try-with-resources for all native objects) is well-documented and consistently applied in tests. The build pipeline reliably produces signed, multi-platform artifacts and publishes them to Maven Central with minimal manual intervention.

Areas worth attention going forward: expanding test coverage beyond happy-path numeric correctness, promoting the Linux amd64 build to an explicit matrix entry so it carries its own CI signal, broadening JDK 21/25 coverage beyond Linux, fleshing out the one-line `CONTRIBUTING.md`, and tying `code-smells.md` items back to lint rules, CI checks, or tracked tests so the audit stays load-bearing rather than purely advisory.
